## Docker Images

### Image 1 — `newsletter-base` (shared layer)

Installs Python 3.13 + `uv` + all dependencies (no browser). Used as the base
for agent and frontend.

```dockerfile
# docker/base/Dockerfile
FROM python:3.13-slim

# Install uv
COPY --from=ghcr.io/astral-sh/uv:latest /uv /uvx /usr/local/bin/

WORKDIR /app

# Install deps (layer-cached unless pyproject.toml changes)
COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-dev --no-install-project

# Copy source
COPY src/ ./src/
COPY config/ ./config/

ENV PATH="/app/.venv/bin:$PATH"
```

### Image 2 — `newsletter-agent`

Extends base. No browser needed.

```dockerfile
# docker/agent/Dockerfile
FROM newsletter-base
CMD ["python", "-m", "src.agent.agent"]
```

### Image 3 — `newsletter-pipeline`

Needs camoufox (Firefox runtime). Larger image (~1.5 GB).

```dockerfile
# docker/pipeline/Dockerfile
FROM python:3.13-slim

# Install Firefox runtime deps
RUN apt-get update && apt-get install -y \
    libglib2.0-0 libnss3 libatk1.0-0 libatk-bridge2.0-0 \
    libcups2 libdrm2 libxkbcommon0 libxcomposite1 libxdamage1 \
    libxfixes3 libxrandr2 libgbm1 libasound2 \
    && rm -rf /var/lib/apt/lists/*

COPY --from=ghcr.io/astral-sh/uv:latest /uv /uvx /usr/local/bin/

WORKDIR /app

COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-dev --no-install-project

# Pre-download camoufox browser binary
RUN python -c "import camoufox; camoufox.download()"

COPY src/ ./src/
COPY config/ ./config/

ENV PATH="/app/.venv/bin:$PATH"
CMD ["python", "-m", "src.knowledge.pipeline"]
```

### Image 4 — `newsletter-frontend`

Extends base, adds Streamlit + FastAPI.

```dockerfile
# docker/frontend/Dockerfile
FROM newsletter-base
EXPOSE 8501 8000
CMD ["bash", "-c", "uvicorn src.frontend.server:app --port 8000 & streamlit run src/frontend/app.py --server.port 8501"]
```

## Key Challenges

### 1. Shared data (SQLite + ChromaDB)

The agent reads articles from SQLite/ChromaDB; the pipeline writes to them.
Both need access to `data/`.

**Options:**

| Approach | Complexity | Notes |
|---|---|---|
| **Single PVC (ReadWriteOnce)** | Low | Works for personal use — only one writer at a time. Requires agent and pipeline to schedule on the same node. |
| **PostgreSQL + pgvector** | Medium | Migrate `raw_store` → Postgres, `vector_store` → pgvector extension. Proper multi-pod access. Best for production. |
| **ChromaDB server mode** | Medium | Run ChromaDB as a separate `StatefulSet`. Agent and pipeline connect over HTTP. SQLite stays on PVC. |

**Recommendation for now:** single PVC (ReadWriteOnce), collocated on one node.
Schedule migration to Postgres + pgvector as a follow-up if concurrency becomes
a problem.

### 2. Gmail OAuth token (`token.json`)

`token.json` is rewritten on every token refresh — it cannot live in a read-only
Secret. Options:

| Approach | Notes |
|---|---|
| **Mount on PVC** | Store `creds/` directory on the same PVC as `data/`. Simple but couples credential storage to data volume. |
| **Writable Secret via External Secrets Operator** | Syncs from AWS Secrets Manager / GCP Secret Manager and writes back on refresh. Complex setup. |
| **Workload Identity (GKE)** | Replace user OAuth entirely with a GCP service account key. No token refresh problem. Only works on GCP. |
| **Init container pre-auth** | Mount token from Secret, copy to writable emptyDir at startup. Works but token expires between restarts. |

**Recommendation:** PVC for `creds/`. Copy the initial `token.json` +
`credentials.json` into the PVC during first-time setup via a one-off Job.

### 3. camoufox / Firefox in pipeline

The browser binary download is ~300 MB. Bake it into the pipeline image at
build time (see Dockerfile above) so containers start without downloading.

## Kubernetes Resources

### Directory layout

```
k8s/
  namespace.yaml
  pvc.yaml                  # shared data + creds volume
  secrets/
    env-secret.yaml         # template — real values in .gitignore
  configmap.yaml            # newsletters.yaml, speech_replacements.yaml
  agent/
    deployment.yaml
  pipeline/
    cronjob.yaml
  frontend/
    deployment.yaml
    service.yaml
    ingress.yaml
  kustomization.yaml
```

### Namespace

```yaml
# k8s/namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: newsletter
```

### PersistentVolumeClaim

```yaml
# k8s/pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: newsletter-data
  namespace: newsletter
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 5Gi     # articles.db + chroma index + creds
```

### Secrets

```yaml
# k8s/secrets/env-secret.yaml  (template — do not commit real values)
apiVersion: v1
kind: Secret
metadata:
  name: newsletter-env
  namespace: newsletter
stringData:
  LIVEKIT_URL: ""
  LIVEKIT_API_KEY: ""
  LIVEKIT_API_SECRET: ""
  DEEPGRAM_API_KEY: ""
  OPENAI_API_KEY: ""
  GROQ_API_KEY: ""       # if using open-source model plan
```

### ConfigMap

```yaml
# k8s/configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: newsletter-config
  namespace: newsletter
data:
  newsletters.yaml: |
    medium:
      label: Medium Daily Digest
      ...
  speech_replacements.yaml: |
    - pattern: '\bRAG\b'
      replacement: R A G
    ...
```

Mount ConfigMap over `config/` instead of baking it into the image — allows
newsletter config changes without rebuilding.

### Agent Deployment

```yaml
# k8s/agent/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: newsletter-agent
  namespace: newsletter
spec:
  replicas: 1
  template:
    spec:
      containers:
      - name: agent
        image: ghcr.io/<user>/newsletter-agent:latest
        envFrom:
        - secretRef:
            name: newsletter-env
        volumeMounts:
        - name: data
          mountPath: /app/data
        - name: creds
          mountPath: /app/creds
        - name: config
          mountPath: /app/config
      volumes:
      - name: data
        persistentVolumeClaim:
          claimName: newsletter-data
      - name: config
        configMap:
          name: newsletter-config
```

> **Tip:** Scale to 0 replicas when not in use to avoid LiveKit idle charges.
> Add a `poe agent-up` / `poe agent-down` shortcut that runs `kubectl scale`.

### Pipeline CronJob

```yaml
# k8s/pipeline/cronjob.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: newsletter-pipeline
  namespace: newsletter
spec:
  schedule: "0 7 * * *"     # 07:00 UTC daily (adjust to your timezone)
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: pipeline
            image: ghcr.io/<user>/newsletter-pipeline:latest
            envFrom:
            - secretRef:
                name: newsletter-env
            volumeMounts:
            - name: data
              mountPath: /app/data
            - name: creds
              mountPath: /app/creds
            - name: config
              mountPath: /app/config
          volumes:
          - name: data
            persistentVolumeClaim:
              claimName: newsletter-data
```

### Frontend Deployment + Service + Ingress

```yaml
# k8s/frontend/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: newsletter-frontend
  namespace: newsletter
spec:
  replicas: 1
  template:
    spec:
      containers:
      - name: frontend
        image: ghcr.io/<user>/newsletter-frontend:latest
        ports:
        - containerPort: 8501   # Streamlit
        - containerPort: 8000   # FastAPI token endpoint
        envFrom:
        - secretRef:
            name: newsletter-env
        volumeMounts:
        - name: data
          mountPath: /app/data
        - name: creds
          mountPath: /app/creds
---
# k8s/frontend/service.yaml
apiVersion: v1
kind: Service
metadata:
  name: newsletter-frontend
  namespace: newsletter
spec:
  selector:
    app: newsletter-frontend
  ports:
  - name: streamlit
    port: 80
    targetPort: 8501
  - name: api
    port: 8000
    targetPort: 8000
---
# k8s/frontend/ingress.yaml  (requires ingress-nginx or similar)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: newsletter-frontend
  namespace: newsletter
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  tls:
  - hosts: [newsletter.yourdomain.com]
    secretName: newsletter-tls
  rules:
  - host: newsletter.yourdomain.com
    http:
      paths:
      - path: /api
        backend:
          service:
            name: newsletter-frontend
            port:
              number: 8000
      - path: /
        backend:
          service:
            name: newsletter-frontend
            port:
              number: 80
```

## Cluster Options

### Option A — Hetzner VPS + k3s *(Cheapest, recommended for personal use)*

Single VPS running k3s (lightweight Kubernetes distribution).

| Spec | Cost |
|---|---|
| CX22 (2 vCPU, 4 GB RAM) | ~€4/mo |
| CX32 (4 vCPU, 8 GB RAM) | ~€9/mo |

**Setup:**
```bash
# Install k3s
curl -sfL https://get.k3s.io | sh -

# k3s bundles: containerd, CoreDNS, Traefik ingress, local-path storage
# No separate ingress controller needed
```

**Pros:** Cheapest, full control, persistent storage works out of the box with
local-path provisioner. Good for a single-user personal project.
**Cons:** No managed control plane (you manage upgrades); single node = single
point of failure; need to set up DNS + TLS manually (cert-manager).

---

### Option B — DigitalOcean Kubernetes (DOKS)

Managed k8s, minimal ops overhead.

| Spec | Cost |
|---|---|
| 1× s-2vcpu-4gb node | ~$24/mo |
| 1× s-4vcpu-8gb node | ~$48/mo |

**Pros:** Managed control plane (free), built-in load balancer, easy volume
provisioning, good docs.
**Cons:** 3–5× more expensive than Hetzner for equivalent specs.

---

### Option C — GKE Autopilot (Google Cloud)

Fully managed, serverless-style k8s. You pay per pod CPU/memory, not per node.

**Cost:** ~$0.05–0.15/day for a single agent pod + daily CronJob.
No charge when pods are scaled to 0.

**Pros:** Best for bursty workloads (agent scaled to 0 most of the day);
Workload Identity solves the Gmail OAuth problem elegantly (replace OAuth user
flow with a service account); no node management.
**Cons:** GCP-specific patterns (Workload Identity); slightly more complex initial
setup; storage options add cost.

---

### Option D — Local cluster (k3d / kind / minikube)

Run k8s locally on the Mac for development and testing before pushing to cloud.

```bash
# k3d (Docker-based k3s)
brew install k3d
k3d cluster create newsletter --port "8501:80@loadbalancer"
```

**Best for:** Testing k8s manifests locally before deploying to a cloud cluster.
Not a production hosting option.

## CI/CD — GitHub Actions

Build and push images on merge to `main`; apply manifests automatically.

```yaml
# .github/workflows/deploy.yml
name: Build & Deploy
on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: docker/login-action@v3
      with:
        registry: ghcr.io
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}
    - name: Build & push agent
      uses: docker/build-push-action@v6
      with:
        context: .
        file: docker/agent/Dockerfile
        push: true
        tags: ghcr.io/${{ github.repository }}/agent:latest
    - name: Build & push pipeline
      uses: docker/build-push-action@v6
      with:
        context: .
        file: docker/pipeline/Dockerfile
        push: true
        tags: ghcr.io/${{ github.repository }}/pipeline:latest

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - name: Apply manifests
      env:
        KUBECONFIG_DATA: ${{ secrets.KUBECONFIG }}
      run: |
        echo "$KUBECONFIG_DATA" | base64 -d > kubeconfig
        kubectl --kubeconfig kubeconfig apply -k k8s/
        kubectl --kubeconfig kubeconfig rollout restart \
          deployment/newsletter-agent -n newsletter
```

## Implementation Steps

| Step | Task | Effort |
|---|---|---|
| 1 | Write Dockerfiles (base, agent, pipeline, frontend) | 2 hr |
| 2 | Test images locally with `docker compose` | 1 hr |
| 3 | Write k8s manifests (namespace, PVC, Secret template, ConfigMap) | 1 hr |
| 4 | Write agent Deployment + pipeline CronJob manifests | 1 hr |
| 5 | Set up cluster (Hetzner + k3s recommended) | 1–2 hr |
| 6 | First-time creds bootstrap (copy `token.json` to PVC via Job) | 30 min |
| 7 | Write GitHub Actions workflow | 1 hr |
| 8 | Frontend Deployment + Service + Ingress + TLS | 1 hr |

**Total: ~8–10 hours end-to-end.**

## Recommended Starting Order

1. **Hetzner CX32 + k3s** for the cluster — cheapest, batteries included.
2. **Docker Compose first** — validate images locally before touching k8s.
3. **Pipeline CronJob** — least stateful component, good first k8s deploy.
4. **Agent Deployment** — add after pipeline is confirmed working on cluster.
5. **Frontend** — last, after agent is stable and frontend code is written.
6. **GitHub Actions** — add once manual deploys are repeatable.
